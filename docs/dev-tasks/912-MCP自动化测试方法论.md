## 一、方法论总览

### 1.1 测试方法论定位

```
┌─────────────────────────────────────────────────────────────┐
│  S6 测试报告 = 标准动作清单 + 客观数据 + 风险记录            │
│     ↑                                                        │
│     └─ 本文（012 文档）定义：标准动作是什么、如何拿客观数据  │
│        ↓                                                     │
│  当期 S6 报告按本文模板产出，并补充本次特有的验证项          │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 三层验证模型

| 层级 | 内容 | 工具 | 输出 |
|---|---|---|---|
| **L1 静态验证** | TypeScript 编译、构建、VS Code 诊断 | `npx vue-tsc --noEmit` / `npx vite build` / `GetDiagnostics` | 退出码、错误数 |
| **L2 运行时验证** | DOM 状态、布局尺寸、事件响应 | `mcp_chrome-devtools` / `mcp_playwright` | DOM 快照、计算属性、截图 |
| **L3 真机验证** | 触摸手势、拖拽手感、性能感知 | 真机/物理设备 | 用户主观反馈（标记为待办） |

> **核心原则**：L1 必须 100% 自动通过；L2 尽量自动化；L3 标注"待真机"。

### 1.3 MCP 工具替代关系

| 场景 | 原 mcp_playwright | 现 mcp_chrome-devtools | 备注 |
|---|---|---|---|
| 页面导航 | `browser_navigate` | `navigate_page` | 等价 |
| 元素点击 | `browser_click` | `click(uid)` | chrome-devtools 需 uid 精确引用 |
| 截图 | `browser_take_screenshot` | `take_screenshot` | 等价 |
| DOM 状态查询 | `browser_evaluate` | `evaluate_script(function)` | 等价 |
| 多断点 | `browser_resize` | `resize_page(w, h)` | **chrome-devtools 实际值=DPR 缩放后，需注意** |
| 文件上传 | `browser_file_upload` | `upload_file(uid, filePath)` | chrome-devtools 需 uid 指向可触发 file chooser 的元素 |
| 控制台消息 | `browser_console_messages` | `list_console_messages(types=[...])` | chrome-devtools 参数结构不同 |
| 元素结构快照 | `browser_snapshot` | `take_snapshot` | 元素 uid 系统等价 |
| TouchEvent 模拟 | `browser_evaluate` 自构造 | **不支持** | 需 mcp_playwright 或真机 |

> **结论**：chrome-devtools 可独立完成 90% 的 UI 测试，触摸交互需 mcp_playwright 或真机补全。

## 三、L1 静态验证（标准动作）

### 3.1 TypeScript 编译验证

```bash
npx vue-tsc --noEmit
```

**判定标准**：
- 退出码 0 → 通过
- 退出码非 0 → 列出错误，区分"本次引入"与"预存在"（参考 010 经验教训 §2.1）
- **预存在错误可接受，但必须在 S6 报告中标注"非本次引入"**

### 3.2 构建验证

```bash
npx vite build
```

**判定标准**：
- 退出码 0 → 通过
- 记录构建时长（监控性能）
- 记录产物大小（chunks > 1500 kB 警告属预存在）

### 3.3 VS Code 诊断（可选）

通过 `GetDiagnostics` 工具获取 LSP 诊断，对比 `vue-tsc` 结果。

### 3.4 静态验证记录模板

```markdown
## 编译/静态验证
- 命令: npx vue-tsc --noEmit
- 结果: 通过（退出码 0，0 错误）/ 失败（X 个错误，其中 Y 个为预存在）
- 命令: npx vite build
- 结果: 通过（exit code 0，56.28s, 3140 modules）
```

## 四、L2 运行时验证（核心方法论）

### 4.1 验证步骤（标准流程）

```
1. 准备阶段
   - resize 到目标视口
   - navigate 到应用
   - 导入测试数据
   - 等待地图/组件初始化完成

2. DOM 状态采集
   - 读取目标元素的 getBoundingClientRect
   - 读取关键 computed style
   - 验证关键类名/属性存在性

3. 交互验证
   - click 目标元素
   - 验证响应（DOM 变化、事件触发、控制台日志）
   - 验证无副作用（其他元素未受意外影响）

4. 数据采集
   - 控制台错误/警告（list_console_messages）
   - 网络请求（list_network_requests）
   - 截图（take_screenshot）
   - 性能数据（如需）

5. 数据对比
   - 实测值 vs 预期值
   - 列出偏差项
   - 评估是否阻塞
```

### 4.2 DOM 状态检查清单（按元素类型）

#### 4.2.1 布局容器（顶栏/底栏/侧栏）

| 检查项 | 测量方法 | 预期值 | 判定 |
|---|---|---|---|
| 元素存在性 | `document.querySelector('.selector')` | 非空 | ✅/❌ |
| 可见性 | `getComputedStyle(el).display` | ≠ 'none' | ✅/❌ |
| 位置 | `getBoundingClientRect().top/left/right/bottom` | 在视口内 | ✅/❌ |
| 尺寸 | `getBoundingClientRect().width/height` | 符合预期 | ✅/❌ |
| overflow | `document.scrollWidth > document.clientWidth` | false | ✅/❌ |
| z-index | `getComputedStyle(el).zIndex` | 不被遮挡 | ✅/❌ |

#### 4.2.2 交互组件（按钮/输入/Tab）

| 检查项 | 测量方法 | 预期值 |
|---|---|---|
| 可点击 | element 存在 + display≠none + cursor:pointer | ✅ |
| 点击响应 | click 后 DOM 变化（ref/e1 vs e5） | 状态切换 |
| 键盘可访问 | Tab 键能 focus | ✅（额外） |
| ARIA 属性 | role/aria-* 属性正确 | ✅（额外） |

#### 4.2.3 数据展示组件（列表/树/表格）

| 检查项 | 测量方法 | 预期值 |
|---|---|---|
| 渲染项数 | `document.querySelectorAll('.item').length` | = 预期 |
| 激活态类名 | `.active` 类存在 | ✅ |
| 数据绑定 | 显示的文本与传入数据一致 | ✅ |

#### 4.2.4 地图组件（OpenLayers）

| 检查项 | 测量方法 | 预期值 |
|---|---|---|
| 容器存在 | `.map-container` 存在 | ✅ |
| 可见尺寸 | `getBoundingClientRect()` 有 width/height | > 0 |
| 控件加载 | 缩放/比例尺/版权等控件可见 | ✅ |
| 事件注册 | `map.on('click', ...)` 已绑定 | 通过 evaluate 验证 |

### 4.3 验证脚本模板

```javascript
// evaluate_script 通用验证函数
() => {
  const checks = {
    layout: {
      topBar: { selector: '.top-bar', expectHeight: 40 },
      navArea: { selector: '.mobile-nav-area', expectVisible: true },
      mapContainer: { selector: '.map-container', expectVisible: true },
    },
    state: {
      sheetExists: !!document.querySelector('.bottom-sheet'),
      overlayVisible: !!document.querySelector('.sheet-overlay'),
    },
    console: {
      // 通过 list_console_messages 单独获取
    },
  }

  const result = {}
  for (const [group, items] of Object.entries(checks.layout)) {
    const el = document.querySelector(items.selector)
    if (!el) {
      result[group] = { exists: false }
      continue
    }
    const rect = el.getBoundingClientRect()
    const display = getComputedStyle(el).display
    result[group] = {
      exists: true,
      display,
      width: rect.width,
      height: rect.height,
      top: rect.top,
      hasOverflow: items.expectHeight && Math.abs(rect.height - items.expectHeight) > 1,
    }
  }
  return { ...result, ...checks.state }
}
```

### 4.4 5 断点全覆盖验证

**强制动作**（§11.3.2 V1/V2/V3）：

```
1. resize_page 切换 5 个断点（顺序：xs → sm → md → lg → xl）
2. 每个断点执行：
   a. navigate_page（强制刷新，避免缓存干扰）
   b. upload_file 测试数据
   c. 等待地图初始化（~2s）
   d. 采集关键元素的 boundingRect
   e. 验证 document.scrollWidth ≤ document.clientWidth
3. 输出矩阵表（必填项）：
   | 断点 | 视口设置 | 实测视口（DPR） | hasHOverflow | topBar尺寸 | contentArea尺寸 | mapContainer尺寸 | navArea可见 |
```

**实测视口与设置值不一致的常见原因**：
- Edge 浏览器 DPR 缩放（设置 390 → 实测 500 左右）
- 窗口最大化限制
- **处理方式**：以实测视口为准，但需在报告中标注 DPR/缩放比例

### 4.5 控制台错误/警告检查

```javascript
// mcp_chrome-devtools: list_console_messages
// types: ["error"] → 找错误
// types: ["warn"] → 找警告
```

**判定标准**：
- error 数量 = 0 → 通过
- error 数量 > 0 → 列出每条错误，区分本次引入/预存在
- 警告：标注即可，不阻塞

**已知预存在警告**（基线）：
- `instanceId prop type Number vs String`（一期引入）

### 4.6 截图证据规则

**何时截图**：
- 关键状态切换（开/关抽屉、激活/未激活 Tab）
- 5 断点布局对比

- Bug 复现现场

**保存位置**：

## 九、常见失败模式与应对

### 9.1 MCP 工具连接断开

**症状**：`mcp error: MCP tool is not found` 或 `Could not connect to msedge`

**应对**：
1. 检查 Edge 浏览器是否在运行（`Get-Process msedge`）
2. 切换 MCP（chrome-devtools ↔ playwright）
3. 重新 navigate_page 初始化连接
4. **不要**：重复重试同一个工具（已记录为反模式）

### 9.2 file upload 路径权限

**症状**：`browser_file_upload` 报"path not in allowlist"

**应对**：
1. 将测试文件复制到 `C:\Users\YuanYu\.playwright-mcp\` 后再上传
2. **不要**：尝试修改路径或绕过权限

### 9.3 视口尺寸与设置值不一致

**症状**：设置 390x844，实测 500x667

**应对**：
1. 以实测视口为准，但报告中标注 DPR 缩放
2. 验证内容用相对值（百分比）而非绝对值
3. 调整断言：`getBoundingClientRect().width / innerWidth < 1`（不溢出）

### 9.4 元素 click 失败："did not become interactive"

**症状**：`click` 报"element did not become interactive within timeout"

**应对**：
1. 用 `take_snapshot` 重新获取元素 uid（uid 可能已变化）
2. 用 `evaluate_script` 查找并直接 `el.click()` 绕过 Playwright 的可交互检查
3. 确认元素未被遮罩层覆盖

### 9.5 控制台警告干扰判断

**症状**：出现大量"既有问题"的警告，难以判断本次改动

**应对**：
1. 在 S0 阶段就建立"基线警告清单"（参考 §L3 §4.5）
2. 报告中只对**新增**警告做判断
3. 警告不阻塞通过，错误才阻塞
