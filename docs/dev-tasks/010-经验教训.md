> 评估基准：`geo-compacter.md` V2.0
> 整理日期：2026-06-25
> 方法论：返工复盘 + 防返工自检清单 + 知识沉淀
> 范围：第一期（布局统一）+ 第二期（Bottom Sheet + FAB）执行过程中的返工、未一次性达到预期的情景

## 一、返工复盘方法论

### 1.1 方法论框架

```
返工识别 → 归因分析 → 教训提炼 → 自检清单 → 知识沉淀
   ↓         ↓         ↓         ↓         ↓
返工环节   原因类型   通用规律   前置检查   写入技能
```

### 1.2 返工归因分类

| 归因类型 | 说明 | 典型表现 |
|---|---|---|
| **方案缺陷** | 方案设计阶段未充分考虑场景/约束 | 场景错配、需求理解偏差 |
| **执行失误** | 实施过程中操作错误 | 合并丢失、文件遗漏 |
| **验证不足** | 验证覆盖面不全或方法不当 | 仅测主断点、未真机验证 |
| **外部依赖** | 工具/环境/平台限制 | PowerShell 语法差异、MCP 无法模拟触摸 |

## 二、第一期返工复盘

### 2.1 合并丢失返工（执行失误）

**返工环节**：第一期成果从 `feature/phase1-layout-unify` 合并到 `main`（b5e4496）时，8 个文件的一期改动丢失。

**归因分析**：
- 直接原因：合并策略选择不当，可能使用了 squash 或未正确处理冲突
- 根本原因：合并后未执行 `git diff main..feature --stat` 验证完整性

**教训提炼**：
1. **合并完整性验证**：分支合并后必须 diff 验证所有改动文件已进入目标分支
2. **合并策略明确化**：功能分支合并使用 `--no-ff` 保留合并记录，便于追溯
3. **关键文件清单**：合并前列出预期改动文件清单，合并后逐项核对

**防返工自检清单**：
- [ ] 合并前：`git diff <target>..<source> --stat` 输出预期文件清单
- [ ] 合并后：`git log --oneline <target> -5` 确认合并 commit 存在
- [ ] 合并后：`git diff <target>~1..<target> --stat` 确认改动文件数符合预期
- [ ] 合并后：编译验证（`vue-tsc --noEmit`）+ 启动验证

### 2.2 导入弹窗响应式未完成（方案缺陷 - 低优先级）

**返工环节**：第一期诊断 1.4 标注的导入弹窗 `max-width` 适配未在第一期实施。

**归因分析**：
- 直接原因：第一期聚焦布局架构，低优先级项被推迟
- 根本原因：方案未明确"推迟项"的归属期次

**教训提炼**：
1. **推迟项归属明确化**：方案中标注"推迟"的项目必须明确归属期次，避免遗漏
2. **验收清单完整性**：每期验收清单应包含本期诊断的所有项，未完成项需显式标注

## 三、第二期返工复盘

### 3.1 FAB 场景错配返工（方案缺陷 - 重大）

**返工环节**：第二期设计 FAB 承载绘制工具，实施完成后用户反馈"已经进入数据查看阶段你还弄个fab让我绘制？完全是为了fab来左的。完全没用"，需移除 FAB。

**归因分析**：
- 直接原因：方案设计阶段未充分评估 FAB 在"数据查看"场景的适用性
- 根本原因：缺少"场景适用性自检"环节，将 Material Design 规范机械套用

**教训提炼**：
1. **场景优先于规范**：组件规范（如 FAB 56dp）必须服从于业务场景，不能为用规范而用规范
2. **场景适用性自检**：方案设计阶段必须回答："此组件在该场景下是否增加了用户价值？"
3. **功能冗余识别**：新引入组件前，先检查现有组件是否已覆盖该功能（MapControlPanel 已有绘制工具）

**防返工自检清单（方案设计阶段）**：
- [ ] **场景价值自检**：新组件在该场景下是否提供不可替代的用户价值？
- [ ] **功能冗余自检**：现有组件是否已覆盖该功能？新组件是否造成重复？
- [ ] **风格一致性自检**：新组件风格是否与整体设计语言一致？（FAB Material vs Element Plus）
- [ ] **用户路径自检**：用户在该场景下的核心任务是什么？新组件是否服务于核心任务？

### 3.2 Flex 布局溢出返工（执行失误 - 3 个问题）

**返工环节**：移除 FAB 后发现 3 个布局问题：
1. 底部 Tab 栏宽度只有 99px（应为全宽）
2. 内容区高度为 0（地图不可见）
3. Tab 栏高度溢出 1px

**归因分析**：
- 直接原因：Flex 布局嵌套陷阱
  - `flex: 1 1 100%` 在 column 父容器中被内容撑开
  - `flex-basis` 百分比在不同方向父容器中歧义
  - `height: 56px + border-top: 1px` 在 content-box 模型下总高 57px
- 根本原因：缺少"Flex 布局自检清单"

**教训提炼**：
1. **Flex 子项显式约束**：flex 子项必须有显式的 `flex-shrink` / `flex-grow` / `flex-basis`，避免默认值歧义
2. **固定高度区必加 overflow**：`flex-shrink: 0` + 固定高度的容器必须加 `overflow: hidden`，防止子项撑开
3. **border-box 默认化**：所有布局组件默认 `box-sizing: border-box`，避免 border/padding 撑开
4. **flex-basis 百分比慎用**：在嵌套 flex 容器中，`flex-basis: 100%` 的含义取决于父容器方向，优先用 `width: 100%`

**防返工自检清单（Flex 布局实施后）**：
- [ ] 每个 flex 子项是否有显式 `flex` 简写或 `flex-grow/shrink/basis`？
- [ ] 固定高度区（顶栏/底栏/Tab栏）是否 `flex-shrink: 0` + `overflow: hidden`？
- [ ] 所有布局容器是否 `box-sizing: border-box`？
- [ ] 嵌套 flex 容器中，`flex-basis` 百分比的方向是否与父容器 `flex-direction` 一致？
- [ ] 是否在 5 断点动态验证布局无溢出？

### 3.3 断点验证不完整返工（验证不足）

**返工环节**：二期最初仅验证 xl(1280)，用户反馈 390x844/430x932 溢出后才补全 5 断点验证。

**归因分析**：
- 直接原因：验证策略偏向"主断点"，忽略移动端关键断点
- 根本原因：缺少"5 断点全覆盖"的验证规范

**教训提炼**：
1. **5 断点全覆盖**：响应式改造必须 5 断点（xs/sm/md/lg/xl）全部动态验证，不能仅测"主断点"
2. **移动端优先验证**：移动端断点（xs/sm）是溢出高发区，应优先验证
3. **验证矩阵文档化**：每期验收必须输出 5 断点验证矩阵表

**防返工自检清单（响应式改造验证阶段）**：
- [ ] xs(375) 动态验证：topBar/content/navArea 边界 = 视口边界？
- [ ] sm(600) 动态验证：同上
- [ ] md(768) 动态验证：左面板显示/隐藏切换正确？
- [ ] lg(1024) 动态验证：左面板宽度 280px？
- [ ] xl(1280) 动态验证：左面板宽度 320px？
- [ ] 每个断点输出：topBar/content/map/navArea 的 getBoundingClientRect

### 3.4 PowerShell heredoc 失败（外部依赖）

**返工环节**：`git commit -m "$(cat <<'EOF'...EOF)"` 在 PowerShell 中解析失败。

**归因分析**：
- 直接原因：PowerShell 不支持 Bash heredoc 语法
- 根本原因：跨平台 shell 脚本未考虑语法差异

**教训提炼**：
1. **跨平台命令适配**：Windows 环境使用 PowerShell，需避免 Bash 专属语法
2. **多行 commit message 方案**：使用 `git commit -F <file>` + 临时文件方式

**防返工自检清单（Windows 环境命令）**：
- [ ] 多行文本：使用临时文件 + `-F` 参数，而非 heredoc
- [ ] 路径分隔符：使用 `/` 或确保 PowerShell 兼容
- [ ] 引号转义：PowerShell 双引号转义规则与 Bash 不同

### 3.5 GisMobileNav 未按方案改名（方案缺陷 - 低优先级）

**返工环节**：方案要求 `GisMobileNav.vue` 改名为 `GisMobileTabBar.vue`，实际保留原名。

**归因分析**：
- 直接原因：实施时未严格对照方案组件变更表
- 根本原因：方案与实施不一致时未在文档中说明

**教训提炼**：
1. **方案实施对照**：实施完成后必须对照方案组件变更表逐项核对
2. **偏离方案记录**：任何偏离方案的决策必须在文档中说明原因

## 四、通用教训总结

### 4.1 方案设计阶段教训

| 教训 | 适用场景 | 自检问题 |
|---|---|---|
| 场景优先于规范 | 引入新组件 | 新组件在该场景下是否提供不可替代的价值？ |
| 功能冗余识别 | 引入新组件 | 现有组件是否已覆盖该功能？ |
| 风格一致性 | 引入新组件 | 新组件风格是否与整体设计语言一致？ |
| 推迟项归属 | 多期改造 | 推迟项是否明确归属期次？ |

### 4.2 执行实施阶段教训

| 教训 | 适用场景 | 自检问题 |
|---|---|---|
| 合并完整性验证 | 分支合并 | 合并后 diff 是否包含所有预期改动？ |
| Flex 子项显式约束 | Flex 布局 | 每个 flex 子项是否有显式 flex 属性？ |
| border-box 默认化 | 布局组件 | 所有布局容器是否 box-sizing: border-box？ |
| 方案实施对照 | 实施完成 | 是否对照方案组件变更表逐项核对？ |

### 4.3 验证阶段教训

| 教训 | 适用场景 | 自检问题 |
|---|---|---|
| 5 断点全覆盖 | 响应式改造 | xs/sm/md/lg/xl 是否全部动态验证？ |
| 移动端优先验证 | 响应式改造 | 移动端断点（xs/sm）是否优先验证？ |
| 验证矩阵文档化 | 响应式改造 | 是否输出 5 断点验证矩阵表？ |
| 真机依赖项标注 | 触摸交互 | MCP 无法验证的项是否标注"待真机"？ |

### 4.4 跨平台阶段教训

| 教训 | 适用场景 | 自检问题 |
|---|---|---|
| 跨平台命令适配 | Windows 环境 | 是否避免 Bash 专属语法？ |
| 多行 commit message | Git 提交 | 是否使用 -F + 临时文件方式？ |

## 五、三期开工前必读

进入第三期前，以下教训必须转化为三期执行的硬性约束：

1. **方案设计阶段**：三期涉及手势增强，必须先回答"每个手势在该场景下是否提供不可替代的价值？"
2. **执行实施阶段**：三期涉及 Web Worker 和虚拟滚动，必须对照方案组件变更表逐项核对
3. **验证阶段**：三期涉及面板拖拽触控，必须 5 断点全覆盖动态验证 + 真机触摸验证
4. **跨平台阶段**：三期提交时使用 `git commit -F` + 临时文件方式
