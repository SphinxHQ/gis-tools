## 问题描述

用户口述了"数据集—坐标系—地图"三者关系的 5 条规则，并指出当前代码在坐标转换后存在"数据坐标系、地图坐标系有误"的 bug。经代码定位发现：

1. **R3、R4 被 handleResetCrs 违反**：重设坐标系时直接 `updateDataset` 改原数据集 crs，导致地图 projection 跟着变，且未生成新数据集
2. **datasetActiveDataMap 缓存陈旧**：数据集删除/更新时缓存未同步清理，可能导致 GisMapSlot 渲染到旧 crs 的数据
3. **GisDataTransformer 版本恢复脆弱**：`reloadVersionsData()` 失败时 activeVersion 仍指向空 data 的版本，且 `crsVersions.value[0].data = data` 直接引用 props.data，存在引用污染风险

## 根因分析

### 根因 1：handleResetCrs 语义与规则不一致

**位置**：[GisDataTransformer.vue:332-371 handleResetCrs](../../src/components/data/GisDataTransformer.vue)

```typescript
const handleResetCrs = (crs: CrsInfo) => {
  // ...
  activeVersion.value.data.crs = new GisCrs(crs.epsgCode)  // 改活跃版本 data 的 crs
  // ...
  if (activeId.value) {
    // ...
    updateDataset(activeId.value, activeVersion.value.data)  // ❌ 改了原数据集的 crs
  }
}
```

**根因**：重设坐标系被实现为"修改当前数据集的 crs 声明"，而非"基于当前数据生成一份声明了新 crs 的新数据集"。这与 `addTransformVersion`（转换创建新数据集）的语义不一致，违反 R3（地图坐标系不变性）和 R4（重设必然生成新数据）。

**不是症状**：这不是 UI 显示问题，而是业务语义实现错误。仅改 UI 无法解决。

### 根因 2：datasetActiveDataMap 缓存生命周期未对齐数据集生命周期

**位置**：[GisData.vue:156-186](../../src/components/data/GisData.vue)

```typescript
const datasetActiveDataMap = ref<Map<string, GisDataInfo>>(new Map())

const getDatasetActiveData = (entryId: string, entryData: GisDataInfo): GisDataInfo => {
  return datasetActiveDataMap.value.get(entryId) ?? entryData  // 优先用缓存
}

const handleActiveDataChange = (data: GisDataInfo, transformChain: number[]) => {
  // ...
  if (activeId.value) {
    datasetActiveDataMap.value.set(activeId.value, data)  // 缓存活跃数据
  }
}
```

**根因**：
- `datasetActiveDataMap` 的 key 是 `entryId`，但未在 `removeDataset`/`updateDataset` 时同步清理或更新
- `removeDataset` 在 store 中执行，GisData.vue 无法直接感知（除非 watch datasets）
- 当数据集 A 被转换 → 缓存 A 的 active data（crs=目标）→ 切换到新数据集 B → 切回 A → GisMapSlot 用缓存的 active data（crs=目标）渲染，但 A 的 entry.data.crs 仍是原 crs → 数据坐标系与地图坐标系不一致

**不是症状**：这是缓存生命周期管理缺失，不是渲染问题。

### 根因 3：GisDataTransformer 版本恢复逻辑脆弱

**位置**：[GisDataTransformer.vue:222-258](../../src/components/data/GisDataTransformer.vue)

```typescript
watch(() => props.data, (newData) => {
  // ...
  if (newEntry?.transformVersions && newEntry.transformVersions.length > 1) {
    crsVersions.value = newEntry.transformVersions.map(snap => ({
      // ...
      data: new GisDataInfo(),  // 空数据
      // ...
    }))
    crsVersions.value[0].data = data  // origin 用当前 data（引用赋值）
    reloadVersionsData()  // 重新计算转换后版本的 data
    activeVersionName.value = newEntry.activeVersionName || 'origin'
  }
})
```

**根因**：
- `reloadVersionsData()` 内部调用 `transformData()`，可能因 proj4 defs 未注册、坐标超出范围等失败，失败时版本的 `data` 仍是空的 `new GisDataInfo()`，但 `activeVersionName` 仍指向该版本 → 显示空数据
- `crsVersions.value[0].data = data` 是引用赋值，后续 `reloadVersionsData()` 中 `item.data = GisDataInfo.clone(originData.value)` 会重新克隆，但 origin 版本（idx=0）的 data 仍是 props.data 的引用 → 如果 props.data 被外部修改（如 updateDataset），crsVersions[0].data 跟着变
- `activeVersionName.value = newEntry.activeVersionName || 'origin'`：如果保存的 activeVersionName 对应的版本 data 转换失败，会显示空数据

**不是症状**：这是版本恢复逻辑的健壮性问题，不是单一 bug。
