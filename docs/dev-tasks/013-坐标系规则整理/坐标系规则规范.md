> **规则来源**：用户口述（2026-06-26）
> **适用范围**：gis-tools 项目全前端数据处理与地图可视化流程

> **维护任务**：`docs/dev-tasks/013-坐标系规则整理/`
> **关联文档**：`docs/dev-tasks/003-数据生命周期/S1-需求分析.md`、`docs/dev-tasks/911-遗留问题.md`

## 一、引言

本规范定义"数据集—坐标系—地图"三者关系的 5 条核心规则（R1~R5）。所有涉及数据集切换、坐标转换、坐标重设、要素编辑、地图渲染的代码实现，必须严格遵守本规范。

规则编号一旦分配，不再变更。代码关键位置应通过注释引用规则编号，便于追溯。

## 二、术语定义

| 术语 | 定义 |
|------|------|
| **数据集（Dataset）** | `GisDataStore.datasets` 数组中的一个 `GisDataSetEntry`，包含 `id`、`name`、`data`（GisDataInfo）、`sourceId`、`transformVersions` 等 |
| **CRS** | 坐标参考系统，通过 `epsgCode` 标识，如 EPSG:4490（CGCS2000）、EPSG:3857（Web Mercator） |
| **地图实例（Map Instance）** | 每个 `datasetId` 对应一个独立的 `GisMapSlot` 组件实例，通过 `v-show` 切换显示 |
| **活跃数据（Active Data）** | 数据集当前显示的数据（可能经过 CRS 转换），由 `GisDataTransformer` 管理，缓存于 `datasetActiveDataMap` |
| **转换路径（Transform Chain）** | 数据集从 origin 到当前 CRS 的转换历史，持久化于 `transformVersions`，由 `CrsVersionSnapshot` 数组构成 |
| **转换（Transform）** | 坐标数值从源 CRS 转换到目标 CRS（如 EPSG:4490 → EPSG:3857），坐标值发生实际变化 |
| **重设（Reset）** | 仅声明数据的新 CRS，不进行坐标数值转换（用于修正原数据 crs 声明错误的场景） |
| **要素编辑（Feature Edit）** | 修改 `GisDataInfo.features`（增删改几何/属性），不涉及 crs |

## 三、5 条规则正文

### R1：地图随数据集切换

**陈述**：任何切换数据集的行为都要伴随地图切换。

**动机**：每个数据集有独立的 CRS 和要素集合，必须由独立的地图实例承载其投影与渲染。共用地图实例会导致投影切换、要素覆盖、状态污染。

**实现位置**：
- [GisData.vue](../../src/components/data/GisData.vue) `<gis-map-slot v-for="entry in datasets" :key="entry.id" :visible="entry.id === activeId">`
- 通过 `v-show` 切换显示，不销毁重建（与 project_memory 一致）

**违反后果**：地图投影与数据 crs 不匹配；要素渲染到错误投影；切换数据集后状态丢失。

**验证方法**：
1. 加载数据集 A（EPSG:4490）→ 地图显示 A 的投影
2. 加载数据集 B（EPSG:3857）→ 地图切换为 B 的投影
3. 切换回 A → 地图恢复 A 的投影，A 的要素仍可见

---

### R2：数据-坐标系与地图深度绑定

**陈述**：数据-坐标系 与地图深度绑定。地图的 `projection` 直接来源于数据集 `data.crs.epsgCode`。

**动机**：地图投影必须严格反映数据声明的 CRS，不允许任何中间层"修正"或"覆盖"。这是数据-地图一致性的根本保证。

**实现位置**：
- [GisMapSlot.vue](../../src/components/gismap/GisMapSlot.vue) `const epsgCode = computed(() => props.data?.crs?.epsgCode)`
- `:options="{ projection: epsgCode }"` 传给底层 GisMap
- [GisData.vue](../../src/components/data/GisData.vue) `getDatasetActiveData(entry.id, entry.data)` 返回活跃数据，其 `crs.epsgCode` 即地图投影

**违反后果**：地图投影与数据 crs 不一致 → 要素渲染位置错误、坐标拾取错位、转换结果不可信。

**验证方法**：
1. 数据集 crs 为 EPSG:4490 → 地图 projection 为 EPSG:4490
2. 修改数据集 crs（仅可通过 R4 的转换/重设生成新数据集实现）→ 新数据集的地图 projection 为新 crs
3. 原数据集 crs 不变 → 原地图 projection 不变

---

### R3：地图坐标系不变性

**陈述**：任何情况下，与数据对应的地图坐标系均不会发生变化。

**动机**：数据集一旦创建，其 crs 即不可变。地图坐标系跟随数据集 crs，因此也不可变。任何"修改数据集 crs"的操作都必须通过 R4 的"生成新数据集"实现，原数据集保持不变。

**实现位置**：
- [gisDataStore.ts](../../src/composables/gisDataStore.ts) `updateDataset(id, data)` 仅用于要素编辑场景（R5），不应改 crs
- [GisDataTransformer.vue](../../src/components/data/GisDataTransformer.vue) `handleResetCrs` 必须调用 `addDataset` 创建新数据集，禁止 `updateDataset` 修改原数据集 crs

**违反后果**：原数据集 crs 被修改 → 原地图 projection 被切换 → 已渲染要素位置错误；转换路径历史失真；用户预期被打破。

**验证方法**：
1. 数据集 A（EPSG:4490）→ 重设为 EPSG:3857
2. 检查 A 的 crs 仍为 EPSG:4490（不变）
3. 检查 A 的地图 projection 仍为 EPSG:4490（不变）
4. 检查新数据集 B 出现在列表，crs 为 EPSG:3857
5. 切换到 B → 地图 projection 为 EPSG:3857；切换回 A → 地图 projection 恢复 EPSG:4490

---

### R4：转换/重设生成新数据

**陈述**：坐标【转换】、【重设】必然生成一份新数据。原数据集保持不变。

**动机**：转换和重设是"派生"操作，不应破坏原始数据。新数据集作为独立节点存在，便于对比、回溯、撤销。

**实现位置**：
- [GisDataTransformer.vue](../../src/components/data/GisDataTransformer.vue) `addTransformVersion` 调用 `addDataset(transformedClone, activeSourceId.value)` —— 转换生成同源新数据集
- [GisDataTransformer.vue](../../src/components/data/GisDataTransformer.vue) `handleResetCrs` 调用 `addDataset(resetClone, activeSourceId.value)` —— 重设生成同源新数据集
- `sourceId` 与原数据集相同，便于在数据源列表中归组

**违反后果**：原数据被覆盖 → 无法对比转换前后；转换路径历史失真；用户无法回退到原始数据。

**验证方法**：
1. 数据集 A（EPSG:4490）→ 转换为 EPSG:3857
2. 检查 A 仍在列表，crs 为 EPSG:4490，要素坐标为原始值
3. 检查新数据集 B 出现在列表，crs 为 EPSG:3857，要素坐标为转换值
4. 重设场景同上，仅坐标数值不发生转换

---

### R5：要素变化与地图自身坐标系无关

**陈述**：要素变化与地图自身坐标系无关。要素编辑只改 `features`，不改 `crs`，更不影响地图 projection。

**动机**：要素编辑是数据内容的修改，不涉及坐标系声明。地图 projection 由 crs 决定，与 features 无关。

**实现位置**：
- [GisFeatureTree.vue](../../src/components/data/GisFeatureTree.vue) `promptUpdateOrSaveAs` 调用 `updateDataset(activeId, props.data)` 时，`props.data.crs` 保持不变，仅 `features` 变化
- [GisDataSlot.vue](../../src/components/data/GisDataSlot.vue) 编辑模式下的要素修改，通过 `updateDataset` 同步，不改 crs

**违反后果**：要素编辑意外修改 crs → 地图 projection 切换 → 已渲染要素位置错误；用户编辑要素导致地图投影跳变。

**验证方法**：
1. 数据集 A（EPSG:4490）→ 进入要素编辑模式
2. 修改要素几何（移动顶点、删除要素）
3. 退出编辑 → 选择"更新当前数据"
4. 检查 A 的 crs 仍为 EPSG:4490（不变）
5. 检查地图 projection 仍为 EPSG:4490（不变）
6. 检查要素已更新（几何变化）

## 四、规则关系图

```
┌─────────────────────────────────────────────────────────────────┐
│                      数据集（Dataset）                          │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │  data: GisDataInfo                                        │  │
│  │  ├── crs.epsgCode  ◄──── 决定地图 projection（R2）         │  │
│  │  └── features      ◄──── 要素编辑仅改此（R5）              │  │
│  └───────────────────────────────────────────────────────────┘  │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │  transformVersions: CrsVersionSnapshot[]                   │  │
│  │  └── 转换路径历史（R4：转换/重设生成新数据集）              │  │
│  └───────────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────────┘
                  │
                  │ R1: 切换数据集 → 切换地图实例（v-show）
                  │ R3: 数据集 crs 不可变 → 地图 projection 不可变
                  ▼
┌─────────────────────────────────────────────────────────────────┐
│                    地图实例（Map Instance）                     │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │  GisMapSlot                                                │  │
│  │  ├── datasetId  ◄──── 与数据集 1:1 绑定                    │  │
│  │  ├── data       ◄──── getDatasetActiveData(entry.id, ...)  │  │
│  │  └── epsgCode   ◄──── computed(props.data.crs.epsgCode)（R2）│  │
│  └───────────────────────────────────────────────────────────┘  │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │  GisMap（底层）                                            │  │
│  │  └── projection = epsgCode  ◄──── 来自 GisMapSlot           │  │
│  └───────────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────────┘
```

## 五、违反规则的处理流程

1. **登记**：在 [911-遗留问题.md](../911-遗留问题.md) 追加 bug 登记，标注违反的规则编号
2. **根因分析**：在对应任务 S1 文档中定位根因，明确违反位置
3. **修复**：在对应任务 S2/S3 中设计修复方案，必须引用本规范规则编号
4. **回归验证**：按规则的"验证方法"逐条验证
5. **状态更新**：911 中 bug 状态从 ⏳ 待处理 → ✅ 已解决
