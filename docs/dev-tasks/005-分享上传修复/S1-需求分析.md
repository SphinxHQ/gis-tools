# S1: 需求分析 — 分享上传修复

## 需求来源
- S0 文档: `S0-原始需求确认.md`
- 已确认需求摘要: 三个修复项——①移动端上传文件类型过滤 ②安卓分享面板无 PWA ③PWA 图标视觉调整

## 问题描述

### 问题1：移动端上传只显示媒体文件
用户在手机安装 PWA 后，点击"上传文件"时，系统文件选择器只显示图片/视频/音频等媒体文件，无法选择 .geojson/.shp/.dxf 等空间数据文件。

### 问题2：安卓分享面板无 PWA
用户在安卓手机上从其他 App 分享文件时，系统分享面板中没有出现已安装的 GIS Tools PWA。

### 需求3：PWA 图标视觉调整
`pwa-icon.svg` 的线条宽度、节点半径需增加 30%，中间填充不透明度绝对值加 0.30。

## 根因分析

### 问题1 根因
`src/components/data/GisDataReader.vue:226` 的 `el-upload` 组件**未设置 `accept` 属性**。

- 桌面端浏览器：无 accept 时显示完整文件系统，用户可手动选择任意文件，故桌面端无此问题
- 移动端浏览器（Android Chrome/Edge）：无 accept 时，文件选择器默认进入"媒体"分类，只显示图片/视频/音频。必须通过 accept 属性显式声明可接受的文件类型（MIME 或后缀），移动端才会显示"文件"分类让用户浏览文件系统选择非媒体文件

### 问题2 根因
经核查，构建产物 `dist/manifest.webmanifest` 中**已包含** `start_url: "/gis-tools/"` 和 `scope: "/gis-tools/"`（VitePWA 基于 `base: '/gis-tools/'` 自动补充），且 `share_target` 字段完整。

真正根因是：**用户安装 PWA 时使用的是 004 之前的旧 manifest（无 share_target 字段）**。已安装的 PWA 会缓存首次安装时的 manifest，新增 share_target 后：
- Android Chrome：通常会在 PWA 更新时重新读取 manifest，但更新时机不确定（可能需数小时或需手动触发）
- 最可靠方式：卸载旧 PWA → 重新访问站点 → 重新安装 → 系统才会注册新的 share_target

显式在 `vite.config.ts` 中补充 `start_url` 和 `scope` 虽非根因修复，但可：
1. 消除配置歧义，让源码自文档化
2. 避免依赖 VitePWA 自动推断行为（版本升级时行为可能变化）
3. 与 share_target.action 路径保持显式一致

### 需求3 根因
纯视觉参数调整，无根因。当前数值：
- stroke-width: 16 → 目标 16 × 1.3 = 20.8 → 取 21
- circle r: 20 → 目标 20 × 1.3 = 26
- fill rgba alpha: 0.15 → 目标 0.15 + 0.30 = 0.45

## 现状分析（已有基础设施 vs 缺失部分）

### 已有
- `GisDataReader.vue` 上传组件完整，`handleFileChanged` 处理逻辑正常
- `DataFormat.ts` / `GisDataInfo.ts` 解析器支持 GeoJson/WKT/WKB/Exchange/CSV/ShapeFile/ShapeZip/EXF/DXF
- `vite.config.ts` VitePWA 配置基本完整，share_target 已配置
- `pwa-icon.svg` 图标文件存在

### 缺失
- `el-upload` 缺少 `accept` 属性（问题1）
- `vite.config.ts` manifest 缺少显式 `start_url` 和 `scope`（问题2 配置补全）
- `pwa-icon.svg` 视觉参数需调整（需求3）

## 影响范围

### 问题1 影响
- 文件: `src/components/data/GisDataReader.vue`
- 模块: 数据导入对话框
- 场景: 移动端用户上传空间数据文件
- 桌面端: 无影响（accept 不限制桌面端文件选择，仅作为过滤器）

### 问题2 影响
- 文件: `vite.config.ts`
- 模块: PWA manifest 配置
- 场景: 移动端系统分享面板
- 注意: 配置修改后需用户重装 PWA 才能生效

### 需求3 影响
- 文件: `public/pwa-icon.svg`
- 模块: PWA 图标（源文件）
- 注意: `pwa-192x192.png` / `pwa-512x512.png` 是由 SVG 生成的 PNG，本次仅改 SVG 源文件，PNG 需用户后续重新生成（或本次用工具转换）

## 约束条件
- 不破坏桌面端现有上传行为
- accept 属性需覆盖解析器支持的所有格式，避免遗漏
- manifest 修改不破坏现有 PWA 安装
- SVG 修改后尺寸仍为 512×512，不改变 viewBox

## 验证结论
- 构建产物 manifest.webmanifest 已含 start_url/scope（VitePWA 自动补充），确认问题2 根因是旧 PWA 缓存而非配置缺失
- 解析器支持的格式已从 GisDataType 枚举 + getDataType 逻辑确认

## 最小改动假设（微创基线）

### 最少需改动的文件/函数
1. `src/components/data/GisDataReader.vue` — el-upload 添加 accept 属性（1 处）
2. `vite.config.ts` — manifest 添加 start_url 和 scope 字段（2 行）
3. `public/pwa-icon.svg` — 修改 stroke-width、circle r、fill alpha（3 处数值）

### 明确不需要改动的区域
- `DataFormat.ts` / `GisDataInfo.ts`（解析器不改）
- `sw-share-target.js` / `useShareReceiver.ts`（004 已交付）
- `GisData.vue`（不改）
- `handleFileChanged` 逻辑（不改，仅改 accept 属性）

### 可复用能力
- 复用 GisDataType 枚举推导 accept 后缀列表
- 复用 VitePWA 现有配置框架

### 选择该最小范围的理由
三个问题相互独立，各自有明确的单点修复位置，无需跨模块改动。问题2 的配置补充虽非根因修复，但用户已确认要求补充配置 + 告知重装。
