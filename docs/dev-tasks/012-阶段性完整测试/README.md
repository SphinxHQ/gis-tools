## 三、三层验证模型速查

| 层级 | 内容 | 工具 | 通过标准 |
|---|---|---|---|
| **L1 静态验证** | TS 编译、构建、LSP 诊断 | `npx vue-tsc --noEmit` / `npx vite build` / `GetDiagnostics` | 退出码 0，无新增错误 |
| **L2 运行时验证** | DOM 状态、布局尺寸、事件响应 | `mcp_chrome-devtools` / `mcp_playwright` | 实测值符合预期，无水平溢出 |
| **L3 真机验证** | 触摸手势、拖拽手感、性能感知 | 真机/物理设备 | 用户主观反馈通过 |

## 四、使用方式

### 4.1 测试前准备（必做）

参考 [912 §2.1](../912-MCP自动化测试方法论.md#21-环境就绪检查) 标准动作：

```
[ ] 1. 启动 dev server：npx vite --port 5174 --host
[ ] 2. 检查 MCP 工具连接（chrome-devtools / playwright）
[ ] 3. 准备测试数据（简单/复杂/边界）
[ ] 4. 识别本期必测项（来自各文件 ✅ 标记项）
```

### 4.2 执行顺序建议

```
1. 先跑 01-L1静态验证（必须 100% 通过才能继续）
       ↓
2. 再跑 04-5断点响应式矩阵（xl → xs 逐断点验证）
       ↓
3. 逐模块执行 02-L2功能验证 + 03-L2-UX-UI验证
       ↓
4. 最后核对 06-防返工自检清单
       ↓
5. 汇总 05-L3真机验证待办（交付用户真机测试）
```

### 4.3 测试结果记录

每项测试用以下标记记录结果：

- ✅ 通过
- ❌ 失败（附现象描述）
- ⏳ 待真机（L3 项）
- ⚠️ 阻塞（附原因）
- ➖ 不适用（附理由）

### 4.4 风险接受规范

参考 [912 §七](../912-MCP自动化测试方法论.md#七风险接受记录规范)：禁止把"无法测"等同于"通过"，必须显式标注跳过原因与接受依据。
