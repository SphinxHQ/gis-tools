# S1: 需求分析 — 布局统一与架构重构

## 需求来源

- S0 文档: `S0-原始需求确认.md`
- 已确认需求摘要: 按 `ux-renovation-plan.md` 第一期方案，消除双套布局硬切换，建立 5 级断点体系，统一为弹性布局骨架

## 问题描述

当前 `GisData.vue` 使用 `v-if="showLeftPanel"` 在两套完全不同的 DOM 结构间硬切换：
- `desktop-layout`：左面板 + 拖拽手柄 + 地图 + 状态栏
- `mobile-layout`：地图 + 状态栏 + 底部导航

这违反了 `geo-compacter.md` 的三条硬性约束：
1. **禁止 `v-if` 硬切换两套布局结构**——当前代码正是如此
2. **同一套 DOM 结构**——当前是两套
3. **单一代码路径**——虽然业务逻辑没有 if 分支，但 DOM 结构的硬切换导致组件实例在断点变化时被销毁重建

同时，`useBreakpoint.ts` 缺少 `xs`（<576px）断点，仅有 4 级（sm/md/lg/xl），不符合 5 级断点规范。

## 根因分析

| 问题 | 根因 |
|------|------|
| 双套布局硬切换 | `resolveBreakpoint` 返回 `sm` 时 `PANEL_WIDTH[sm] = 0`，导致 `showLeftPanel = false`，触发 `v-if` 切换到 mobile-layout |
| 缺少 xs 断点 | `resolveBreakpoint` 函数中 `return 'sm'` 覆盖了所有 <768px 的宽度，没有区分 <576px |
| 顶栏高度反直觉 | `.is-mobile .top-bar { height: 36px }` 硬编码为 36px，比桌面 40px 更矮 |
| 弹窗无响应式 | `el-dialog width="96%"` 是固定百分比，无 `max-width` 限制 |

## 现状分析

### useBreakpoint.ts（src/composables/useBreakpoint.ts）

```typescript
// 当前：4 级断点
export type Breakpoint = 'xl' | 'lg' | 'md' | 'sm'

function resolveBreakpoint(width: number): Breakpoint {
  if (width >= 1280) return 'xl'
  if (width >= 1024) return 'lg'
  if (width >= 768) return 'md'
  return 'sm'  // <768px 全部归为 sm
}

// isMobile 仅匹配 sm
const isMobile = computed(() => globalBreakpoint.value === 'sm')
// showLeftPanel 基于 panelWidth > 0
const showLeftPanel = computed(() => PANEL_WIDTH[globalBreakpoint.value] > 0)
```

**缺失**：无 `xs` 断点；`isMobile` 不覆盖 `xs`；无 `showBottomNav`。

### GisData.vue 模板（src/components/data/GisData.vue L370-430）

```vue
<!-- 有数据时的布局 -->
<template v-else>
  <!-- 桌面端：v-if 硬切换 -->
  <div v-if="showLeftPanel" class="desktop-layout">
    <div class="left-panel">...</div>
    <div class="panel-resize-handle" />
    <div class="content-area">
      <div class="map-container">...</div>
      <gis-data-overview mode="full" />
    </div>
  </div>
  <!-- 移动端：v-else -->
  <div v-else class="mobile-layout">
    <div class="map-container mobile-map">...</div>
    <gis-data-overview mode="compact" />
    <gis-mobile-nav />
  </div>
</template>
```

**问题**：两套 DOM 结构完全独立，地图实例（GisMapSlot）在两套布局中各有一份 v-for，断点切换时会导致地图组件销毁重建。

## 约束条件

1. **geo-compacter.md 规范约束**：
   - 禁止 `v-if` 硬切换两套布局结构
   - 禁止在 `<script>` 中编写 `if (isMobile.value) { ... } else { ... }` 分支业务逻辑
   - 差异仅允许存在于 CSS 层和模板层的布局编排
   - 5 级断点：xs(<576) / sm(576-767) / md(768-1023) / lg(1024-1279) / xl(>=1280)

2. **兼容性约束**：
   - 不能破坏 MapDrawer.vue 中对 `isMobile` 的使用
   - GisMobileNav 的 props/events 接口不变
   - GisDataPanel 的 props/events 接口不变

3. **性能约束**：
   - 断点切换时不触发地图组件销毁重建（当前 v-if 硬切换会导致此问题）
   - 统一布局后，地图实例在所有断点下保持挂载

4. **功能约束**：
   - 左面板拖拽 resize 在 ≥1024px 仍可用
   - 左面板折叠/展开功能不变
   - 移动端底部导航功能不变

## 最小改动假设（微创基线）

### 最少需改动的文件/函数

1. **`src/composables/useBreakpoint.ts`**：
   - `Breakpoint` 类型：添加 `'xs'`
   - `resolveBreakpoint` 函数：添加 `<576px → xs` 分支
   - `PANEL_WIDTH` / `PANEL_COLLAPSED_WIDTH`：添加 `xs` 键值
   - `isMobile`：改为 `xs || sm`
   - 新增 `showBottomNav`：`xs || sm`（即 <768px）
   - `showLeftPanel` 保留但语义不变（`panelWidth > 0`，即 md/lg/xl）

2. **`src/components/data/GisData.vue`**：
   - 模板：移除 `v-if="showLeftPanel"` / `v-else` 双套布局，合并为单一 `unified-layout` DOM
   - 模板：左面板、拖拽手柄、GisMobileNav 始终在 DOM 中，通过 CSS 控制显隐
   - 模板：GisDataOverview 的 mode 通过 `:mode="isMobile ? 'compact' : 'full'"` 适配（模板层布局编排，非 script 业务逻辑分支）
   - 样式：`.is-mobile .top-bar` 高度从 36px 改为 48px + safe-area-inset-top
   - 样式：新增 `@media` 规则控制左面板/手柄/底部导航的显隐

3. **`src/components/data/GisDataImportDialog.vue`**：
   - 样式：添加 `@media (min-width: 768px) { .import-dialog { max-width: 640px; } }`

### 明确不需要改动的区域

- `GisMobileNav.vue` 内部逻辑（Tab 切换、抽屉等）——第二期改造
- `GisDataPanel.vue` 内部逻辑——不变
- `MapControlPanel.vue`——不变
- `GisMapSlot.vue` / `GisMapTianditu.vue`——不变
- `GisDataOverview.vue` 内部逻辑——不变（仅 mode prop 值变化）
- `MapDrawer.vue`——需验证 `isMobile` 语义变化的兼容性，但预计不需要改动（`xs+sm` 仍表示移动端）

### 可复用能力

- 现有 `useBreakpoint` composable 的全局监听机制（`ensureResizeListener` / `maybeRemoveResizeListener`）
- 现有 flexbox 布局模式（`desktop-layout` 已使用 flex）
- CSS `@media` 查询（项目已支持）
- `env(safe-area-inset-*)` CSS 函数

### 选择该最小范围的理由

一期改造方案的目标是"布局统一与架构重构"，核心是消除双套布局、建立断点体系。不涉及组件内部逻辑改造（第二期）、手势增强（第三期）、Token 化（第四期）和代码清理（第五期）。3 个文件的改动恰好覆盖 4 项诊断违规，不多不少。
