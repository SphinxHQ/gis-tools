# S1: 需求分析 — 移动端分享接收

## 需求来源
- S0 文档: `S0-原始需求确认.md`
- 已确认需求摘要: 让已安装的 GIS Tools PWA 出现在移动端系统分享面板，接收空间数据文件分享，SW 拦截暂存后页面读取并自动导入为新数据集

## 问题描述
当前 PWA 虽已具备基础能力（vite-plugin-pwa autoUpdate），但无法接收系统分享：
1. manifest 未声明 share_target，系统分享面板不会列出本 PWA
2. Service Worker 无拦截 POST 分享请求的逻辑，静态站点无法处理 POST
3. 页面无读取分享数据并导入的入口

## 根因分析
不是 bug，是功能缺失。根因是 PWA 配置层缺少 share_target 声明 + SW 层缺少 POST 拦截 + 应用层缺少分享数据消费入口。三层缺一不可。

1. manifest 缺 share_target 字段

## 影响范围
- **文件层**:
  - `vite.config.ts`（修改：manifest 加 share_target + workbox 加 importScripts）
  - `public/sw-share-target.js`（新建：SW 拦截逻辑，放 public 因为是静态资源）
  - `src/composables/useShareReceiver.ts`（新建：页面端读取分享数据并导入）
  - `src/components/data/GisData.vue`（修改：onMounted 调用 useShareReceiver 初始化）
- **模块层**: PWA 配置模块、SW 模块、数据导入模块
- **用户场景**:
  - 移动端已安装 PWA 的用户，从文件管理器/其他 App 分享 .geojson/.wkt/.shp/.dxf 等文件 → 系统分享面板出现 GIS Tools → 选中后 PWA 启动并自动导入文件
  - 不影响现有上传/文本/绘制/历史记录导入流程
  - 不影响桌面端使用

## 约束条件
1. **兼容性**: 安卓 Chrome/Edge 完美支持；iOS Safari 16.4+ 支持；iOS 16.3 及以下、微信/QQ 内置浏览器不支持（不兼容场景静默降级，不报错）
2. **安全性**: HTTPS 已满足（部署在 GitHub Pages）；SW 作用域必须覆盖 /gis-tools/share-receiver
3. **性能**: 单文件建议 <50MB（系统限制）；Cache Storage 适合小文件，大文件场景首期不引入 IndexedDB（S0 已明确不包含）
4. **业务规则**: 仅接收空间数据文件（GeoJSON/WKT/SHP/SHP-ZIP/DXF/EXF/Exchange），不接收图片/PDF/Word
5. **子路径约束**: 部署在 /gis-tools/ 子路径，share_target.action 必须是 `/gis-tools/share-receiver`，重定向目标必须是 `/gis-tools/?share=1`
6. **缓存清理**: 处理完分享数据必须删除缓存，避免占用空间和重复读取
7. **URL 清理**: 处理完清除 `?share=1` 参数，避免刷新重复处理

## 验证结论（如有）
- vite-plugin-pwa 1.3.0 支持 `workbox.importScripts` 配置，可引入自定义 SW 脚本（已查文档确认）
- generateSW 模式下，importScripts 脚本在 workbox 初始化后执行，可注册额外 fetch 监听器拦截 POST 请求，与 workbox 自身 fetch 监听器不冲突（workbox 对未匹配路由的请求不调用 respondWith，放行给后续监听器）
- 用户示例代码中 `JSON.stringify(payload)` 含 File 对象会丢失二进制数据，需修正：文件用独立 Response 存储为 Blob，元数据用另一个 Response 存储为 JSON

## 最小改动假设（微创基线）
- **最少需改动的文件/函数**:
  1. `vite.config.ts` VitePWA 配置块：manifest 加 share_target，workbox 加 importScripts: ['sw-share-target.js']
  2. `public/sw-share-target.js`（新建）：fetch 事件监听器，拦截 POST /gis-tools/share-receiver，暂存文件到 caches，303 重定向
  3. `src/composables/useShareReceiver.ts`（新建）：检测 ?share=1，从 caches 读取文件，调用 SimpleDataFormat + addDataSource 导入，清理缓存
  4. `src/components/data/GisData.vue` onMounted：调用 useShareReceiver().init()
- **明确不需要改动的区域**:
  - GisDataReader.vue（复用其 SimpleDataFormat 解析路径，但不修改组件本身）
  - DataFormat.ts 及各 DataFormat 实现（格式识别与解析逻辑不变）
  - gisDataStore.ts（复用 addDataSource，不修改）
  - 路由配置（不新增路由，复用首页 / 路由 + URL 参数判断）
  - App.vue、main.ts（不改入口）
  - 现有 workbox 预缓存策略（不破坏）
- **可复用能力**:
  - SimpleDataFormat.read(content: ArrayBuffer | string) —— 直接解析分享文件
  - useGisDataStore().addDataSource(dataInfo) —— 直接添加为新数据集
  - localDb.add(name, content, meta) —— 持久化到历史记录
  - getDataType(content) —— 自动识别格式
- **选择该最小范围的理由**:
  - 不新增路由：分享重定向到 `/?share=1`，复用首页，避免路由表膨胀
  - 不修改 GisDataReader：直接在 composable 中调用 SimpleDataFormat + addDataSource，与 GisDataReader 的导入路径一致，但避免修改组件内部逻辑
  - 保持 generateSW 模式：不切换到 injectManifest，避免重写预缓存逻辑，仅通过 importScripts 注入分享拦截
