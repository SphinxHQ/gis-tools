# S0: 原始需求确认 — 移动端分享接收

## 用户原始需求

实现移动端 PWA 接收系统分享功能，核心依靠 Web Share Target API，让 PWA 注册进系统分享面板，与其他原生 App 并列出现，用户选中后直接把数据传给应用。

完整接收流程分为「manifest 注册 → Service Worker 拦截 → 页面读取处理」三步。

必须满足的前提：
- 站点必须 HTTPS（本地 localhost 除外）
- 必须是已安装到主屏幕的 PWA
- 必须有合法 manifest 和正常运行 Service Worker
- 浏览器内核支持：安卓 Chrome/Edge、iOS Safari 16.4+；微信/QQ 内置浏览器不支持

部署地址：`https://sphinxhq.github.io/gis-tools/`（满足 HTTPS 要求）

技术方案要点：
1. manifest.json 中添加 share_target 字段，声明接收文件类型、请求方式、接收地址
2. Service Worker 拦截 POST 请求，暂存文件到 Cache Storage（大文件用 IndexedDB），303 重定向到页面
3. 页面加载时判断 URL 分享标记，从缓存读取文件和数据，执行业务逻辑，处理完清理缓存

兼容性：安卓 Chrome/Edge 完美支持；iOS Safari 16.4+ 支持；iOS 16.3 及以下不支持；微信/QQ 内置浏览器不支持；桌面端部分支持。

踩坑点：PWA 必须安装；action 路径要在 SW 作用域内；iOS 限制多；及时清理缓存；单文件建议 <50MB；HTTPS 强制。

## AI 复述后的需求

本次要实现：让已安装的 GIS Tools PWA 出现在移动端系统分享面板中，用户从其他 App 分享空间数据文件（GeoJSON/WKT/SHP/DXF 等）时，可选中本 PWA 接收。接收后文件被 Service Worker 暂存，页面加载时读取并自动导入为新数据集，复用现有 GisDataReader 的导入能力。

技术路径：
1. 在 vite.config.ts 的 VitePWA manifest 中添加 share_target 配置（POST + multipart/form-data，action=/gis-tools/share-receiver）
2. 通过 VitePWA 的 workbox 配置或自定义 SW 注入 fetch 拦截逻辑，暂存分享文件到 Cache Storage，303 重定向到 `/?share=1`
3. 新增分享接收路由或复用首页路由，页面加载时检测 `share=1` 标记，从缓存读取文件，调用现有导入逻辑处理
4. 处理完成后清理缓存，清除 URL 参数

## 需求边界
- 本次包含:
  - manifest 中 share_target 字段配置（接收空间数据文件类型）
  - Service Worker 拦截 POST /gis-tools/share-receiver 并暂存文件
  - 页面读取分享数据并导入为新数据集
  - 缓存清理与 URL 参数清理

  - iOS 16.3 及以下、微信/QQ 内置浏览器的兼容处理
